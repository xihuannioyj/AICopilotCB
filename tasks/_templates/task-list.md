# 当前开发项目任务总表

## 使用规则

- 主 task-list 只保留当前开发窗口内的任务块、角色进度和阻塞信息，不在这里展开字段级实现细节。
- 待排期、治理补漏和长期回填任务统一写入 `tasks/task-backlog.md`。
- 详细实现约束仍分别写在各 feature 目录下的 `backend.md`、`app.md`、`web.md` 与 `api-contract.md`。
- 每个任务块采用两层结构：第一层是任务摘要卡片，记录所属功能目录、任务名称、中文描述、当前状态、预估完成周期、下一动作；第二层是 Agent 跟踪行，记录角色完成状态与遇到问题。
- Agent 接单开始时，先把自己的状态改为“开发中 / 待评审 / 待测试 / 已阻塞”等真实状态；完成收尾时，再改为“已完成”。
- 如果某个 Agent 当前不参与某项任务，状态写“未启用”或“未开始”，不要留空。
- 扩展角色启用时，仍需遵守强制边界：product-manager、doc-manager、test-collector、defect-triage 默认只改 docs/.github，不直接改业务代码。

## 任务卡片机器责任字段

每个任务块除了状态行，还应补一张最小责任卡，供 Agent 和后续脚本统一读取：

| 字段 | 内容 |
|------|------|
| 责任角色 | {architect/backend/flutter/angular/tester/...} |
| 非责任项 | {明确不由当前角色承担的内容} |
| 交接对象 | {下一棒角色} |
| 当前阻塞 | {若不能 done，卡在哪里} |
| 未越界确认 | {例如：未修改其他业务根目录 / 未擅自定稿契约} |

这张责任卡不替代 feature 下的 `backend.md`、`web.md`、`app.md`，而是作为 task-list 总表里的最小摘要，帮助值班和编排快速判断谁该接下一棒。

## 看板选择优先级

- 当前开发窗口内的业务任务：写 `tasks/task-list.md`。
- 尚未进入当前开发窗口的业务候选或长期尾项：写 `tasks/task-backlog.md` 或其子表。
- AI 协同方案、Prompt、Agent、Hook、Runbook、守卫脚本、API 治理等治理专项：写 `tasks/governance-task-list.md`。
- 运行态自动编排、验证结果、批量报告和调度状态：以 `tasks/_runtime/task-index.json` 为准。
- 同一任务只允许一个人工主看板；切换看板时应迁移主状态，不保留双写。
