# 01-通用核心层

## 文档定位

- machine_readable_tag: aicopilotcb-req-core-v2
- 评审对象：AICopilotCB 通用核心层需求细则
- 文档角色：定义跨项目、跨技术栈都必须成立的协同底座
- 预期使用者：方案维护者、架构设计者、接入新项目的 Owner

## 背景与问题

当前协同体系最初从 LiveHome 项目抽离，已经具备多 Agent 协作、任务闭环、Markdown 写入守卫和运行态编排能力，但这些能力仍夹带项目术语、目录路径和历史操作习惯。若没有独立的通用核心层，新项目接入时会重复出现以下问题：

- 把某个项目的例外规则误判为通用规则。
- 新项目只能复制整套控制面，而无法按需裁剪。
- 任务闭环、交接口径、收尾模板等基础协议依赖口头传达。
- Markdown 写入安全、质量守卫、越界阻断等基础能力被散落在多个文档中，难以一致继承。

## 建设目标

- 抽出一套不依赖项目名、业务名和技术栈名也能成立的协同底座。
- 让所有技术栈适配层和项目覆盖层都能继承同一组基础协议。
- 把“先边界、后执行”“先验证、后收尾”的协作方法固定为默认行为，而不是经验性做法。

## 范围

本主题只覆盖以下内容：

- 角色边界定义方法。
- 任务拆分、交接、回流、阻塞表达和完成判定。
- 统一收尾模板与最小验收口径。
- 文档治理在控制面中的地位与同步更新原则。
- 质量守卫的统一责任边界。

## 非目标

- 不定义具体技术栈的构建、测试、联调命令。
- 不定义具体项目的目录、接口前缀、权限 key 或菜单初始化方式。
- 不承载某一业务系统的例外流程。

## 干系角色

- 方案 Owner：决定哪些规则可以升级为通用核心。
- 架构角色：负责把任务闭环和角色边界落到具体协作链路中。
- 文档治理角色：负责把已确认的底座规则同步到说明书、模板和导航入口。
- 质量角色：负责确认验证、回流、阻断与收尾口径未失真。

## 核心需求

### 1. 角色边界模型必须独立存在

- 每个 Agent 必须能独立说明负责项、不负责项、上游输入、下游交接对象。
- 边界模型必须不依赖具体仓库目录名，换项目后仍可复用。
- 任何发现的问题都必须优先判定 owner，再决定修复或回流，禁止以“先修再说”代替归属判断。

### 2. 任务闭环必须是五段式

每个任务至少包含以下五段：

1. 入口：任务来源、适用流程、前置读取对象。
2. 执行：角色各自的处理范围与输出物。
3. 验证：必须执行的校验、检查或最小证明。
4. 回流：发现问题时的归属判断与交接格式。
5. 收尾：已完成项、阻塞项、可选下一步三段式表达。

### 3. 统一收尾模板必须作为硬约束存在

所有协作任务在结束时都必须区分：

- 已完成项。
- 仍阻塞当前目标的未完成项。
- 可选下一步。

该模板的作用是避免把“还能继续优化”误报成“本轮未完成”。

### 4. 文档必须被视为控制面的一部分

- Prompt、Agent、Instructions、Runbook、模板、任务板、守卫脚本都属于协同控制面。
- 规则变更不能只改其中一个载体，必须同步处理会直接影响执行口径的其他载体。
- 文档更新的目标不是写说明，而是维持协同体系的可执行性和可审计性。

### 5. 质量守卫必须有统一责任边界

- 写入成功必须同时满足真实落盘、结构正常和校验通过。
- 守卫脚本是底座能力，不可由单个角色自由跳过。
- 对空写、乱码、结构损坏、断链引用等问题，默认优先修复控制面，再继续后续动作。

## 推荐目录归属

通用核心层后续应主要沉淀到 core/ 下，并至少形成以下稳定对象：

- 协同总则。
- 角色边界基础协议。
- 任务闭环协议。
- 统一收尾模板。
- 越界阻断与风险表达协议。

## 交付物要求

- 一份可直接继承的通用协作总则。
- 一套最小角色边界模板。
- 一套统一收尾模板。
- 一份任务闭环与完成判定说明。
- 一份质量守卫的上层责任说明。

## 验收标准

满足以下条件时，本主题可判定通过：

1. 将 LiveHome 项目名替换为任意其他项目名后，文档主体仍成立。
2. 不引用任何具体技术栈，也能说明清楚协同如何运作。
3. 技术栈适配层和项目覆盖层能明确知道哪些规则必须继承，不允许自行重定义。
4. 收尾口径、回流口径和质量守卫口径可被新项目直接复用。

## 主要风险与约束

- 过度抽象会让规则失去执行抓手，因此仅允许抽取已经验证过的共性协议。
- 若把项目特例提前提升为通用核心，会导致后续多个项目继承错误约束。
- 若不定义最小验收口径，通用核心层会退化为原则汇总，无法支撑真实协作。
