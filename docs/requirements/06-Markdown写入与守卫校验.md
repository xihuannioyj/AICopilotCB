# 06-Markdown写入与守卫校验

## 文档定位

- machine_readable_tag: aicopilotcb-req-markdown-guard-v2
- 评审对象：AICopilotCB Markdown 写入与守卫校验需求细则
- 文档角色：定义协同控制面文档的安全写入与校验闭环

## 背景与问题

协同文档不是普通说明资料，而是控制面的一部分。最近的维护过程中已经出现过以下真实风险：

- 文档被写成 0 字节或 2 字节空白内容。
- notebook-backed Markdown 在工具显示成功后未真实落盘。
- 中文路径导致批量处理漏写或断写。
- notebook 单元结构被破坏，但未被第一时间发现。

因此，这一主题必须被定义为评审级别的强约束，而不是建议项。

## 建设目标

- 建立统一的 判型 -> 写入 -> 校验 -> 失败处置 闭环。
- 把受保护 Markdown 与普通 Markdown 的处理路径彻底分开。
- 确保任何写入失败都能被及时发现、阻断并修复。

## 范围

本主题覆盖：

- 受保护 Markdown 白名单与判型优先级。
- notebook-backed 与普通 Markdown 的写入边界。
- 单文件与多文件的校验方式。
- 写入失败、空写、结构损坏的重试与阻断策略。
- 写入结果的最小可信判断标准。

## 非目标

- 不负责定义业务代码文件的编译或测试策略。
- 不替代 Git、CI 或代码评审系统的职责。
- 不鼓励在未判型时直接进行覆盖式保存。

## 核心需求

### 1. 必须先判型，再选择写入方式

- 先判断目标文件是否属于受保护 Markdown 或高风险路径。
- 若命中白名单或 guard 推荐 notebook-cell，则按 notebook-backed 处理。
- 普通 Markdown 仅允许做局部增量补丁，禁止整文件误覆写。

### 2. 写入成功必须以结果验证为准

以下条件必须同时满足，才可判定写入成功：

- 目标文件已真实落盘。
- 文件非空且编码正常。
- notebook-backed 文件结构未损坏。
- 对应 guard 校验通过。

不得仅以工具返回成功作为完成依据。

### 3. 校验必须紧跟写入

- 单文件改动后立即执行 validate-markdown 或 validate-files。
- 多文件改动结束前必须执行 validate-changed。
- 一旦出现 non_utf8、replacement_char、null_byte、blank_markdown、invalid_notebook_json 等错误，必须先修复，再继续其他动作。

### 4. 必须保留失败处置流程

- 当第一次写入失败时，优先检查判型与写入工具是否匹配。
- 同一路径连续失败不得无限重试，应有明确的最多尝试次数与上报边界。
- 对已知高风险路径，应允许使用更稳定的保底写入方式，但必须保留校验闭环。

### 5. 必须沉淀异常案例

- 对空写、结构损坏、假成功未落盘、中文路径异常等已验证问题，必须沉淀为明确案例。
- 这些案例应进入需求文档或说明性材料，以避免同类错误重复出现。

## 建议交付物

- 受保护 Markdown 白名单。
- 判型顺序说明。
- 写后校验命令清单。
- 失败处置流程。
- 已知异常案例库。

## 验收标准

1. 受保护 Markdown 不再被整文件误覆写。
2. 空写、乱码、结构损坏与假成功落盘问题可以被 guard 及时识别。
3. 写入失败时存在明确重试和上报边界。
4. 多角色协作下，任何人都能按同一闭环维护控制面文档。

## 风险与约束

- 若 guard 只在收尾执行，问题会在多轮修改后才暴露，修复成本显著增大。
- 若不记录已发生的异常案例，团队会反复踩同一类写入问题。
- 若把受保护 Markdown 当普通 Markdown 处理，结构破坏风险始终存在。
