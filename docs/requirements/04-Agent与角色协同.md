# 04-Agent与角色协同

## 文档定位

- machine_readable_tag: aicopilotcb-req-agents-v2
- 评审对象：AICopilotCB Agent 与角色协同需求细则
- 文档角色：定义多 Agent 体系中的职责边界、交接、回流和质量链
- 当前样板编制：8 Agent 主链 + 质量链扩展位

## 背景与问题

多 Agent 协作的失败通常不是因为角色数量不够，而是因为边界不清、交接格式缺失、问题归属不明确，最后演变为多个角色互相越界兜底。要让 AICopilotCB 成为可复用协同底座，必须把角色协同做成可审阅、可替换、可追责的结构化协议。

## 建设目标

- 确保每个 Agent 都能独立说明自己的职责和边界。
- 保证问题先归属、后处理，默认回流优先于越界修复。
- 让交接、回流、收尾和质量采证全部结构化。

## 当前样板角色

- architect
- backend
- angular
- flutter
- product-manager
- doc-manager
- test-collector
- defect-triage

## 非目标

- 不要求所有角色始终同时激活。
- 不要求质量角色直接替代工程角色完成修复。
- 不把一次临时协作方法直接固化为长期角色职责。

## 核心需求

### 1. 每个 Agent 必须在执行前回答四个问题

- 自己负责什么。
- 自己不负责什么。
- 上游输入来自谁。
- 收尾要交给谁。

任何任务文档若未明确这四项，不得直接进入执行。

### 2. 回流优先于越界修复

- 问题属于其他 owner 时，默认回流，不直接越界进入对方目录修改。
- 若问题归属不清，先回流给 architect、product-manager、doc-manager 或质量链判断。
- 任何越界修复都必须被视为异常场景，而不是默认协作方式。

### 3. 交接内容必须结构化

标准 handoff 至少应包含：

- 已完成项。
- 未完成项。
- 阻塞项。
- 下游接手所需信息。
- 当前角色确认未越界的说明。

### 4. 质量链必须保持独立性

- test-collector 负责采证、复现与证据整理。
- defect-triage 负责归因、分类和回流建议。
- 质量链默认不直接进入业务实现目录修复问题。

### 5. 自进化维护能力必须被约束为治理角色，而非全权写入者

- 协同系统自进化能力只能聚焦控制面问题，如 Markdown 写入失败、守卫链失真、任务板职责漂移、角色边界缺失等。
- 这类治理角色默认只负责采集经验、判断复用价值、生成治理建议，不直接覆盖全局制度文件。
- 只有经过验证的规则，才允许升级为通用协议或新角色职责。

## 典型协作链要求

- architect -> backend / frontend：必须交清接口范围、入口闭环、依赖初始化和联调责任。
- backend -> angular / flutter：必须交清真实路由、字段、错误码、样例数据和缺失项。
- frontend -> backend：只回流页面联调发现的问题，不反向接管后端设计责任。
- test-collector -> defect-triage -> owner：质量问题先采证、再归因、后定向回流。

## 建议交付物

- 各 Agent 的机器可读标记模板。
- 标准 handoff 字段模板。
- 典型协作链路示例。
- 常见越界场景与正确回流路径。

## 验收标准

1. 每个 Agent 都可被单独定义、启用、替换或停用。
2. 问题发生时能在短时间内判断 owner，不依赖个人口头记忆。
3. 交接和回流可被文档化审阅，而不是隐藏在聊天上下文里。
4. 质量链与实现链边界清楚，不因紧急情况长期失真。

## 风险与约束

- 如果 handoff 模板缺字段，回流会再次变成口头沟通。
- 如果越界修复被默许为常态，角色边界会快速失效。
- 如果自进化能力缺少安全闸门，协同系统本身可能被未经验证的规则污染。
