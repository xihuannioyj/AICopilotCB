# 02-技术栈适配层

## 文档定位

- machine_readable_tag: aicopilotcb-req-stacks-v2
- 评审对象：AICopilotCB 技术栈适配层需求细则
- 文档角色：定义通用底座如何落到不同技术栈的执行面
- 预期使用者：技术栈负责人、架构角色、新项目接入人

## 背景与问题

通用核心层解决的是协同原则统一，但真实执行时，不同技术栈在目录组织、构建命令、测试方式、调试链路、页面联调路径和常见风险点上差异明显。如果没有单独的技术栈适配层，接入新项目时会出现两类错误：

- 直接复用旧项目规则，导致把不适用的工程约束硬套到新技术栈。
- 每个项目重新定义一套方法，导致底座失去统一性。

## 建设目标

- 在不修改通用核心层的前提下，为目标技术栈建立最小可落地骨架。
- 明确每种技术栈的入口、构建、测试、联调和交接方式。
- 支持后续按技术栈持续细化，而不反向污染通用核心层。

## 范围

本主题覆盖：

- 技术栈目录入口约束。
- 构建、测试、运行的最小入口。
- 技术栈与协作角色的映射方式。
- 页面联调、接口联调、验证链路的差异。
- 技术栈专属模板、README 和后续扩展规划。

## 非目标

- 不在本层定义任何具体业务项目的目录或业务规则。
- 不把单个项目的历史命令直接上升为该技术栈的唯一标准。
- 不在没有样板验证前输出过度细碎的技术细节规范。

## 当前适配对象

- Java
- Python
- Go
- Angular
- Flutter
- Android
- iOS
- WeChat Mini App
- Douyin Mini App

## 核心需求

### 1. 技术栈适配必须继承通用核心层

- 技术栈适配层只允许补充执行差异，不允许重新定义角色边界、回流协议和收尾模板。
- 任一技术栈文档都必须显式说明哪些规则继承自通用核心层。

### 2. 每个技术栈都必须有最小接入骨架

每个 stacks/{tech}/ 至少应包含：

- 技术栈定位与适用范围。
- 推荐读取顺序。
- 构建、运行、测试的最小入口。
- 常见协作角色组合。
- 常见风险点与交接方式。

### 3. 技术栈适配必须支持渐进细化

- 第一阶段只要求 README 骨架可用。
- 第二阶段再补运行手册、模板、校验命令、联调路径和常见故障处理。
- 不允许在尚未验证时一次性扩充出大量低价值模板。

### 4. 前后端与客户端技术栈要区分执行重点

- Java、Python、Go 更关注构建、依赖、接口、测试和运行环境。
- Angular、Flutter、Android、iOS、小程序更关注页面入口、路由、联调、端侧约束和可视化验证。
- 文档必须体现这些差异，而不是给所有技术栈复制同一模板。

### 5. 技术栈适配要为新项目装配提供明确入口

- 新项目接入时，必须能一眼判断应从哪一个 stacks 子目录开始。
- 若项目包含多个技术栈，应能组合多个 stacks 骨架，而不是重新造一套混合规则。

## 建议交付物

- stacks/ 总入口说明。
- 每个目标技术栈一份 README 骨架。
- 技术栈与 Agent 映射关系说明。
- 各技术栈后续补模板的优先级排序。

## 验收标准

1. 每个目标技术栈都有独立入口，且能说明从哪里开始接入。
2. 新项目不需要翻读旧项目历史，即可知道应参考哪份技术栈文档。
3. 技术栈规则不会覆盖通用核心层的基础协议。
4. 相同技术栈的新项目可以复用同一骨架，只补项目覆盖层差异。

## 风险与约束

- 若技术栈适配文档写得过于抽象，接入者仍然需要回头猜构建与验证路径。
- 若把项目经验直接升级为技术栈规则，会造成其他项目接入时的不必要束缚。
- 若技术栈骨架长期只停留在目录占位而无最小入口说明，将失去接入价值。
