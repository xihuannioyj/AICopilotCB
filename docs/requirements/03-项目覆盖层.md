# 03-项目覆盖层

## 文档定位

- machine_readable_tag: aicopilotcb-req-project-overlay-v2
- 评审对象：AICopilotCB 项目覆盖层需求细则
- 文档角色：定义具体项目如何在通用底座上挂接自己的规则
- 当前样板：projects/livehome/

## 背景与问题

协同体系要能跨项目复用，必须允许每个项目保留自己的目录、脚本、任务板、约束和例外规则。如果缺少项目覆盖层，通用层就会被迫承载越来越多项目私货，最终演变为复制一个项目模板到另一个项目，失去通用化价值。

## 建设目标

- 把项目特有规则集中到独立目录管理。
- 确保通用核心层和技术栈适配层保持稳定，不被项目例外反向污染。
- 为新项目接入提供可复制的覆盖层结构。

## 范围

本主题覆盖：

- 项目目录边界。
- tasks 路径与任务板组织方式。
- 项目专属 Prompt、Hook、脚本入口。
- 项目特有 API 约束、权限约束、初始化约束。
- 项目与通用层、技术栈层之间的差异清单。

## 非目标

- 不在本层重写通用协作协议。
- 不把某个项目的全部文档直接复制为新项目唯一标准。
- 不替代技术栈适配层去解释构建或测试的通用差异。

## LiveHome 样板定位

projects/livehome/ 需要承担以下职责：

- 证明三层结构在真实项目上可落地。
- 承接 livehome_admin、livehome_app、livehome_ng 等特有目录边界。
- 承接 LiveHome 的 tasks 组织方式、角色协作口径和特殊约束。
- 作为后续其他项目建立覆盖层时的参考样板，而不是直接复用模板原文。

## 核心需求

### 1. 通用规则与项目规则必须可分离

- 任何写死项目目录、业务术语、路径、脚本、前缀、权限 key 的内容，必须留在项目覆盖层。
- 任何可被其他项目复用的内容，都应优先上移到通用核心层或技术栈适配层。

### 2. 项目覆盖层必须保持轻量

- 项目覆盖层只补真正必要的差异，不重复整套规则。
- 若某条项目规则在两个以上项目中反复出现，应重新评估是否上移为通用层规则。

### 3. 项目覆盖层必须可替换

- 新项目接入时，应通过新增 projects/{project-name}/ 实现，而不是强改现有样板项目。
- 某个项目退出或停止维护，不应影响通用层与其他项目覆盖层的可用性。

### 4. 项目覆盖必须说明与底座的关系

每个项目覆盖层文档至少要说明：

- 继承了哪些通用规则。
- 使用了哪些技术栈骨架。
- 补充了哪些项目私有规则。
- 哪些规则暂时不适合上移为通用规则。

## 建议交付物

- 项目入口说明。
- 项目专属规则清单。
- 项目与通用层的差异说明。
- 项目脚本、任务板、Hook、模板索引。

## 验收标准

1. 换一个项目名后，不需要重写通用层文档。
2. 项目私有规则都能在项目覆盖层集中定位。
3. 新项目接入时可以复制覆盖层骨架，而不是复制整个旧项目控制面。
4. 项目覆盖层不会反向修改通用层定义。

## 风险与约束

- 如果项目覆盖层过厚，说明通用层划分仍然不合理。
- 如果项目覆盖层过薄，接入者会回到通用层中继续猜项目例外。
- 若不持续评估哪些项目规则已经具备复用价值，三层结构会逐渐失衡。
