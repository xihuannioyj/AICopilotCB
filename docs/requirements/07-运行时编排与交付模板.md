# 07-运行时编排与交付模板

## 文档定位

- machine_readable_tag: aicopilotcb-req-runtime-templates-v2
- 评审对象：AICopilotCB 运行时编排与交付模板需求细则
- 文档角色：定义任务板、运行态主数据、交付模板与汇报模板的协同关系

## 背景与问题

仅有角色规则和 Prompt 分流并不足以支撑真实协作。协同体系还必须回答三个问题：任务放在哪里、状态以谁为准、交付和汇报如何输出。若缺少这层设计，系统会出现任务板混写、runtime 数据失真、汇报口径不一致和模板无法复用等问题。

## 建设目标

- 为协同体系建立可运行、可派发、可跟踪、可汇报的控制面结构。
- 区分机器主数据、人工主看板和输出模板的职责。
- 让新项目可以复制整套模板体系，而不是从零定义。

## 范围

本主题覆盖：

- tasks 根目录结构。
- 人工主看板和候选池边界。
- runtime 主数据定位。
- 交付模板与汇报模板体系。
- 数据来源标注与状态解释规则。

## 非目标

- 不要求在本主题中实现实际的 orchestrator 逻辑。
- 不把 runtime 主数据当作人工总结板直接编辑。
- 不允许多个看板共同承担同一任务的实时状态主责。

## 核心对象

### 1. tasks 根目录必须成为统一任务入口

tasks/ 至少应承载：

- task-list.md
- task-backlog.md
- governance-task-list.md
- _runtime/
- _templates/
- feature 级任务目录

### 2. runtime 主数据必须与人工主看板分责

- runtime 主数据负责记录自动编排任务、状态和验证结果。
- 人工主看板只承载 Owner 和协作角色需要直接查看的业务状态。
- 二者职责不同，不能随意互相替代。

### 3. 同一任务只能有一个人工主看板

- 当前开发窗口业务任务进入 task-list.md。
- 候选池任务进入 task-backlog.md。
- 治理专项进入 governance-task-list.md。
- 禁止同一任务在多个人工看板并行双写状态。

### 4. 模板体系必须既机器可读又人可维护

- 业务交付模板至少覆盖 backend、app、web、api-contract。
- 治理与质量模板至少覆盖 product-manager、doc-manager、test-collector、defect-triage。
- 汇报模板至少覆盖日报、周报、里程碑、风险台账、管理摘要与 readiness 检查。

### 5. 汇报必须说明数据来源

- 输出 readiness、日报或阶段结论时，必须说明数据来自 runtime、人工主看板还是候选池。
- 若混用多个来源，必须明确各自负责解释什么，不得隐式拼接后直接下结论。

## 建议交付物

- tasks 目录结构规范。
- 人工主看板职责说明。
- runtime 主数据定位说明。
- 各角色交付模板。
- 日报、周报、readiness、里程碑和风险模板。

## 验收标准

1. 任务、状态、模板和汇报路径清晰可追踪。
2. 不同看板不再混写同一状态。
3. 新项目接入时可以直接复制模板体系并完成裁剪。
4. 汇报结论能够回溯到明确数据来源。

## 风险与约束

- 若 runtime 与人工主看板职责混用，后续自动化能力会持续失真。
- 若模板只有机器可读字段而不便于人工维护，协作角色会绕开模板工作。
- 若汇报不标注数据来源，管理层结论会难以追责和复盘。
