# 08-演进路线与新项目接入

## 文档定位

- machine_readable_tag: aicopilotcb-req-evolution-onboarding-v2
- 评审对象：AICopilotCB 演进路线与新项目接入需求细则
- 文档角色：定义 AICopilotCB 演进为可复用底座的策略，以及新项目如何接入

## 背景与问题

AICopilotCB 当前已经完成复制、隔离、初步整理、需求分拆等基础动作，但距离真正可复用的协同底座仍有差距。主要问题不是资料不够，而是规则抽象层次、接入步骤和演进安全边界还需要进一步固化，否则每接入一个新项目就会重新走一遍人工判断。

## 建设目标

- 保证 AICopilotCB 可以持续独立演进，而不依赖任何单一业务主链。
- 为新项目提供稳定、可审阅、可复制的接入步骤。
- 控制演进过程中的抽象风险与写入风险。

## 范围

本主题覆盖：

- AICopilotCB 的阶段性演进路线。
- 新项目接入的最小装配步骤。
- 接入后的验证动作。
- 演进过程中的主要风险与安全边界。

## 非目标

- 不要求本轮直接完成所有技术栈的深度适配。
- 不要求替换任何既有业务仓库现有协同。
- 不将尚未验证的规则一次性提升为底座标准。

## 分阶段路线

### 阶段 1：复制与隔离

- 从既有控制面来源整理并复制基础文件到 AICopilotCB。
- 保证既有业务主链不受影响。

### 阶段 2：结构化整理

- 建立 README、三层拆分说明、stacks 骨架和首个项目覆盖层样板。
- 将 tasks 迁移到独立根目录，减少 docs 与运行态耦合。

### 阶段 3：需求体系收敛

- 将总需求、拆分方案和相关说明文档下沉为索引 + 专题 requirements。
- 让 requirements 成为唯一评审入口。

### 阶段 4：规则抽象与验证

- 持续把已验证的通用规则上移到 core/ 与 stacks/。
- 保持项目覆盖层尽量薄，并持续回看哪些规则具备复用价值。

### 阶段 5：新项目装配

- 选择目标项目。
- 匹配一个或多个技术栈骨架。
- 新建 projects/{project-name}/ 覆盖层。
- 校准 tasks 路径、Prompt、Agent、守卫脚本与模板。
- 做一次完整写入与校验演练，确认控制面可用。

## 新项目接入最小步骤

1. 确认项目技术栈与协作范围。
2. 选择对应 stacks 子目录作为起点。
3. 继承通用核心层规则。
4. 新建项目覆盖层目录并补项目私有规则。
5. 对齐 tasks 路径、看板路径、守卫脚本和入口 Prompt。
6. 运行一次完整的写入、校验、回流和收尾演练。

## 接入后的最小验证要求

- 能明确分流到正确的 Prompt 入口。
- 能读取到正确的任务板与 runtime 数据。
- 能安全修改受保护 Markdown 并通过 guard。
- 能按角色边界完成一次最小 handoff 与收尾。

## 主要风险

- 把项目私有规则误抽为通用规则。
- 新项目接入时遗漏守卫脚本或校验命令。
- Markdown 写入策略不稳定，导致控制面文档损坏。
- 目录迁移或删减后引用路径未同步。
- 未做最小演练就认定底座可用，导致后续接入连续返工。

## 验收标准

1. AICopilotCB 能继续独立演进而不依赖单一业务主链。
2. 新项目接入有明确步骤，不再依赖口头说明。
3. 三层结构在至少一个样板项目上已验证可用。
4. requirements 体系足以支撑评审，而不是仅做摘要导航。

## 后续优先事项

- 继续把已验证的通用规则落到 core/。
- 为高优先级技术栈补齐第二层 README 之外的最小模板。
- 为第二个真实项目做一次装配验证，检验三层架构是否仍然成立。
